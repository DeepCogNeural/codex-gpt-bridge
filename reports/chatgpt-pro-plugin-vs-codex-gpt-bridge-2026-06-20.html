ChatGPT Pro Plugin vs Codex GPT Bridge

问题：用户日常更想从 Codex 里调用 ChatGPT，尤其希望不用笨重的 @chrome。本报告比较外部 codex-chatgpt-pro-plugin 与本 repo 的 codex-gpt-bridge，并给出融合路线。
TL;DR: 如果主目标是“Codex 调 ChatGPT Pro 网站”，外部 plugin 是更正确的目标线，因为它自己启动并控制专用 Chrome/CDP，不需要 @chrome。但它今天还不能被写成稳定日常默认：安装 smoke、materialized package init 和 live proof 都要先过。本 repo 的强项仍是“ChatGPT UI 调本地 Codex MCP”。

01

结论

先讲人话：两者不是同类产品。一个是 Codex 伸手去叫 ChatGPT Pro，一个是 ChatGPT 伸手来叫本地 Codex。

先修后启用

把 chatgpt-pro 定为 Codex 调 ChatGPT Pro 的目标线；只有安装、打包和 live call 证明通过后，才升级为日常默认。

继续保留

Local Codex Bridge Secure 继续负责 ChatGPT UI 调本地 Codex。它是 MCP/Tunnel 路径，安全边界和工具 schema 已经更清楚。

暂不重写

不要把两个项目硬揉成一个大框架。先修外部 plugin 的安装兼容、materialized package init 和 live 证明，再决定 fork、vendor 或 companion。

02

全方位对比表

核心分歧是方向、控制面和稳定性。表里“更好”只针对你的日常目标：从 Codex 里调用 ChatGPT。

完整对比表
维度codex-chatgpt-pro-plugincodex-gpt-bridge直白判断
主要方向 Codex -> ChatGPT Pro Web UI。Codex 通过 CLI 发 prompt，ChatGPT 网站回答。 ChatGPT UI -> local Codex MCP 是主线；另有 Codex -> OpenAI Responses API 的 ask_chatgpt。 你的日常偏向 Codex 调 ChatGPT，所以外部 plugin 更贴目标；但还需要先通过启用门槛。
是不是 MCP 主能力不是 MCP。它是 Codex plugin + skill + chatgpt-pro CLI。 是 MCP。ChatGPT 看到的是 HTTP MCP app；Codex 侧也有 stdio MCP server。 MCP 不是唯一标准。对 Codex 调 ChatGPT，外部 plugin 的 CLI/browser line 更对题，但当前还不是已验证日常默认。
浏览器控制 自己启动专用 Chrome profile，通过 CDP 控制 127.0.0.1:9222，文本输入走 DOM focus + CDP Input.insertText + DOM button。 不控制 ChatGPT 浏览器。ChatGPT 侧通过 Developer Mode app 和 Secure MCP Tunnel 调本地桥。 外部 plugin 的关键优点就是不用 @chrome，Codex 直接跑 CLI 就能调 ChatGPT 网页。
是否用 ChatGPT Pro 网站能力 是。它依赖用户已登录的 ChatGPT Pro 网站会话，默认选择 live Pro intelligence level。 Codex -> ChatGPT 路径不是网站，是 OpenAI Responses API；需要 OPENAI_API_KEY 和 API 模型访问。 如果你认为网站 Pro 比 API 侧更强，外部 plugin 是唯一对准这点的方案。
上下文传递 可生成 repo-context monofile，带文件树、LOC、文件内容、source map、hash；可上传或 inline，上传需确认。 ChatGPT 通过 codex_read 让本地 Codex 读项目并返回摘要；不会自动把整个 repo 打包上传给 ChatGPT。 外部 plugin 适合让 ChatGPT 直接看大上下文；本 repo 更适合让 Codex 先过滤，再给 ChatGPT 事实。
安全边界 不输入密码/OTP，不读 cookies/session storage；专用 profile；自动生成的 repo context 有 secret path/content scan；但显式上传文件仍要调用者自己 scrub，本质仍是自动化已登录网页账号。 allowed roots、read-only 默认、secret preflight、禁 danger-full-access、write mode owner opt-in；但 allowed root 不是硬 OS sandbox。 两边安全模型不同：plugin 风险在账号/网页自动化；bridge 风险在本地执行边界。
状态与记账 repo-scoped rooms：main/debug/critic/scratch；每次 call 有 receipt、transcript、screenshot、network/console logs。 有 bridge jobs、jobId polling、bridge-created Codex thread registry；状态在本地进程内，重启会丢 job/session registry。 plugin 的 receipts/rooms 更适合长期“advisor/reviewer”工作流，值得吸收。
并发 一个 shared logged-in browser profile，加全局 browser-profile lock 串行化，避免多个 agent 抢同一个 ChatGPT 页面。 HTTP MCP 工具可启动长 Codex job，超过 fast-return 返回 jobId；ChatGPT UI 自动链式 polling 有时会被安全层挡。 plugin 处理“多个 Codex agent 抢 ChatGPT”更具体；bridge 处理“ChatGPT 等本地 Codex”更具体。
稳定性风险 依赖 ChatGPT 网站 DOM、composer、model selector、登录状态。源码有 fail-closed code，但网站变化仍会破。 MCP/tool schema 更稳定；但 ChatGPT UI 安全层、Tunnel 超时、schema refresh 仍会影响使用。 API/MCP 比网页自动化稳；但网页自动化能用到 ChatGPT Pro 产品能力。这是 tradeoff。
当前验证 本次 clone HEAD 0cc5c550，源码和 docs 清楚；npm test 失败在 test:plugin-install，原因是当前 Codex CLI 不接受 marketplace add ... --json。materialized plugin 目录的 init 也会因缺 .codex/skills 失败。 本次 npm run check 通过：TypeScript build + 4 个 Vitest 文件 / 35 tests。 外部 plugin 架构有价值，但拿来就说稳定还太早。本 repo 当前测试状态更稳。
03

融会贯通方案

奥卡姆剃刀：不要造第三个大系统。保留两条链路，把它们定位清楚；外部 plugin 先作为目标线修通，再升级成默认。

可吸收的设计
吸收点来自 plugin放进我们 workflow 的方式
专用浏览器线专用 Chrome profile + CDP，不用 @chrome。作为 Codex 侧默认 ChatGPT Pro advisor line，而不是塞进 MCP bridge。
repo roomsmain/debug/critic/scratch 绑定到 repo。把 advisor 与 reviewer 长期固定为 repo-scoped room，避免每次新开 ChatGPT 丢上下文。
receipts保存 prompt、response、hash、截图、网络和 console 记录。对高风险 planning/review 要求 receipt 路径进入 TASK_STATE 或报告。
repo context自动生成 monofile，带 source map 和 secret scan。用于 architecture/review；普通小任务仍只传 digest/diff，避免上传太多。
browser lock全局 lock 串行化 ChatGPT 网站控制。保留。多个 Codex 子任务可以排队，但不能同时抢同一个 ChatGPT profile。
fail closed登录、composer、response 绑定不清楚时返回稳定错误码。继续采用，尤其是 response.possibly_stale 不能当成功回答。
启用门槛必须通过什么没通过时怎么做
安装兼容修掉 plugin marketplace add ... --json 与当前 Codex CLI 的不兼容，或写 wrapper 兼容新旧 CLI。继续用 source checkout 只读研究，不把它设成默认。
打包路径materialized plugin 里的 bin/chatgpt-pro init 能找到 skills/chatgpt-pro-line/SKILL.md，不再依赖缺失的 .codex/skills。不要从 materialized package 直接推荐安装。
live proofdoctor --warm、人工登录、doctor --live、一次 main advisor call、一次 critic --fresh review call 都成功并落 receipt。Codex 继续用本 repo 的 API-backed ask_chatgpt 做临时模型调用。
上下文安全repo-context 自动扫描通过；显式 --upload-file 只上传人工确认过的窄文件，如 diff.patch 和 test-output.txt。改用摘要，不上传原始 repo 或敏感文件。
日常默认以上全过后，才把 chatgpt-pro 写成 Codex 日常 advisor/reviewer 默认入口。保持“目标线/伴侣工具”定位，不替换当前 MCP bridge。
04

日常流程

两条入口固定下来，后续 agent 不需要猜。注意：chatgpt-pro 是“启用门槛通过后的日常流程”，门槛没过时用 API-backed ask_chatgpt 或继续人工 ChatGPT。

场景用什么固定流程不要做什么
Codex 想问 ChatGPT 做顶层方案 门槛通过后：chatgpt-pro call --alias=main 传 session digest；需要全 repo 时加 --repo-context=auto --confirm-repo-context-upload；要求输出计划、风险、验收标准、下一步给 Codex 的指令。门槛未过时，临时用 ask_chatgpt 或人工 ChatGPT。 不要用 @chrome 人肉操作；不要为小语法问题占用 Pro line。
Codex 做完后要 ChatGPT critical review 门槛通过后：chatgpt-pro call --alias=critic --fresh 上传 diff.patch、test-output.txt 或窄上下文；要求只输出 PASS 或 NEEDS_CHANGES。 不要把 ChatGPT 当 oracle。Codex 仍要用本地文件、测试、日志验证建议。
ChatGPT UI 要调查本地项目 @Local Codex Bridge Secure + codex_read 先 bridge_status；再 codex_read 只传 prompt；如果返回 jobId，显式调用 codex_job_status。 不要让 ChatGPT 猜 cwd/sandbox；不要把 write mode 当默认。
ChatGPT UI 要指挥 Codex 修改 write-mode Local Codex Bridge Secure + codex_run 先用窄 CODEX_GPT_BRIDGE_ROOT 重启 write mode；只让 bridge 内 Codex 做最小编辑；编译/测试/渲染由本地 Codex 执行。 不要让 bridge 内 Codex 同时做大范围修改和长时间验证。

Limitations

没有在这次比较里实际打开已登录 ChatGPT Pro 页面跑 live call；原因是这份任务是架构比较和文档固化，不需要触碰用户真实网页会话。外部 plugin 的网页稳定性仍需单独 live proof。另一个限制是外部 plugin 当前安装 smoke 与本机 Codex CLI 参数不兼容，且 materialized package init 路径假设不一致，要先修。自动 repo context 有 secret scan，但显式上传文件仍需要调用者自己 scrub。
